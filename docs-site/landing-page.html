shipgate

The last gate before production for AI-written code.

AI generates code that compiles, passes linting, and looks correct. But it hallucinates routes, forgets auth, and references env vars that don't exist. shipgate catches what linters can't.

The 3 Lies AI Tells Your Codebase

1. Ghost Routes

AI generates a handler for /api/v1/admin/users but never declares the route in your spec. It compiles. It even returns JSON. But it's a phantom endpoint with no contract.

BLOCK GHOST_ROUTE: Route /api/v1/admin/users not in ISL API spec
  Fix: Add to ISL api block or remove the handler.

2. Ghost Env Vars

AI reads process.env.STRIPE_SECRET_KEY but the variable isn't in your config, your .env.example, or your Truthpack. Deploys will crash silently.

BLOCK GHOST_ENV: STRIPE_SECRET_KEY not declared in ISL config
  Fix: Add to ISL config block or .env.example.

3. Auth Bypasses

AI creates a route handler without auth middleware. Your ISL spec says actor must: authenticated. The code just... doesn't check.

WARN  AUTH_MISSING: Endpoint /api/v1/users accessible without authentication
  Fix: Add auth middleware matching ISL actor constraint.

How It Works

1

Define Intent

Write ISL specs by hand, or describe what you want in plain English. shipgate generates the behavioral contract.

2

Verify Code

shipgate checks every precondition, postcondition, and invariant. Runs 27 policy rules. Cross-references your Truthpack.

3

Gate the Merge

SHIP or NO_SHIP. Exit code 0 or 1. Evidence bundle attached. No ambiguity.

Ship Score

Every verification produces a score from 0 to 100. It measures spec coverage, policy compliance, and verification depth. Not a vibes-based metric — a deterministic calculation you can audit.
